# Post-Event Continuation

Continuation is where 90 percent of civic hackathons die. A sprint creates proof, energy, and public artifacts. It does not magically create maintenance, adoption, funding, legal review, or public-sector capacity.

This section is scaffolding. Actual continuation depends on local follow-through.

## 30-Day Follow-Up Schedule

Day 1: collect final build logs, repos, demos, screenshots, team contacts, and consent to publish.

Day 3: publish project index and thank-you notes.

Day 7: run team check-in, identify dead projects honestly, name promising projects.

Day 14: match promising projects with mentors, maintainers, municipal contacts, NGOs, journalists, or micro-grant leads.

Day 21: review pilot risks: data, privacy, security, legal, maintenance, support, ownership.

Day 30: publish mini impact report and continuation decisions.

## Open-Source Publishing Guide

Use a clear license. Keep secrets out of the repo. Include setup steps, sample data, known limits, and screenshots. Archive dead demos honestly rather than leaving broken links with no explanation.

If a project uses public data, include source links and extraction notes. If a project uses generated data, label it clearly.

## Meeting Template With Municipality

Purpose: decide whether the prototype has a real continuation path.

Agenda:

- Team shows live demo.
- Team states what works and what does not.
- Municipality names the real owner of the problem.
- Group identifies data, legal, security, and maintenance blockers.
- Group decides: close, publish only, pilot exploration, micro-grant, or contract discussion.

No administration receives team work without explicit consent, credit, and a continuation agreement.

## Maintainer Matching

Match teams with maintainers only when the maintainer understands the project limits and team ownership. A maintainer can be a technical volunteer, civic NGO, university lab, newsroom, or municipal innovation unit.
